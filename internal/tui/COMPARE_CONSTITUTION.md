# Comparison Constitution

## Purpose

This document defines what Compare is, what it is not, what operator questions it answers, and the architectural boundaries that govern its implementation. It serves as the specification for the v0.10.x "Explain Comparisons" release.

---

## Identity

### What is Compare?

Compare is Pulse's **decision engine**, not its inspection engine.

Compare answers:

> **"What changed, why did it change, and what should I do next?"**

It takes two completed HTTP investigations and produces a structured analysis of what differs between them. Its output is oriented around **decisions**, not facts ŌĆö regressions, improvements, anomalies, and confidence signals.

Compare exists to eliminate operator hesitation when answering:

* Which result should I trust?
* Did my change fix the problem?
* Is this response worse than the previous one?
* What regressed between the baseline and the candidate?
* Should I investigate further, or is this conclusive?

### What is not Compare?

Compare does NOT answer:

* "What happened during this request?" ŌĆö that is Inspect's job.
* "Show me all response headers." ŌĆö that is Inspect's job.
* "Show me the full response body." ŌĆö that is Inspect's job.
* "What is the raw latency value?" ŌĆö that is Inspect's job.

Compare is not a second Inspect pane. If information is duplicated between Compare and Inspect without adding comparative value, it does not belong in Compare.

Compare is not a record of every field. It is a **selective signal amplifier** ŌĆö it intentionally discards identical information so that only meaningful differences remain visible.

## Guiding Principles

### Principle 1: Differences are primary

The operator should never scan two panes looking for changes.

Compare presents differences first. Similarities are supporting evidence, not the main output. The default view answers "what changed" before "what stayed the same."

### Principle 2: Decision before detail

Compare's rendering hierarchy is:

```
Decision         (Is this better or worse? What action should I take?)
     Ōåō
Difference       (What specifically changed? Status? Latency? Headers? Body?)
     Ōåō
Evidence         (Supporting details for each change)
     Ōåō
Raw data         (Full inspection data, collapsible)
```

The top of the view communicates conclusions. Details are progressively disclosed.

### Principle 3: Conclusions are distinct from facts

Inspect communicates **facts**:

* Status: 200
* Latency: 142ms
* Error: (none)

Compare communicates **conclusions** derived from facts:

* Latency regression (120ms ŌåÆ 350ms, +230ms)
* No functional difference (status unchanged)
* Breaking change: new error introduced

Compare surfaces conclusions. Supporting facts are present but secondary.

### Principle 4: Compare is not Inspect ├Ś 2

Compare must never be implemented as "two Inspect panes side by side." If a section exists in both Compare and Inspect, it must serve different purposes in each context. Duplicated rendering that does not add comparative value must be removed or redesigned.

### Principle 5: Color communicates meaning, not change

Color in Compare must encode semantic direction:

* **Red** ŌĆö worse, regression, degradation
* **Green** ŌĆö better, improvement, fix
* **Yellow** ŌĆö uncertain, requires investigation
* **Muted** ŌĆö unchanged, neutral, supporting context

Never color solely because values differ. "The active result has a different status code" is a fact. "The status regressed from 200 to 404" is a conclusion. Color follows the conclusion, not the fact.

## Operator Workflow

The canonical comparison workflow is:

```
Observe
  Ōåō
Mark Baseline       (c on a result ŌĆö establishes the reference)
  Ōåō
Observe / Run again
  Ōåō
Mark Candidate      (c on a different result ŌĆö establishes the comparison target)
  Ōåō
Compare              (entered automatically after candidate is marked)
  Ōåō
Swap / Replace      (refine the comparison without restarting)
  Ōåō
Clear               (end the comparison session)
```

Key workflow properties:

* **Continuous**: The operator never needs to leave Observe to manage comparisons.
* **Reversible**: Every action (mark, swap, clear) has an inverse or an undo.
* **Discoverable**: Every action appears in the status ribbon when available.
* **Persistent across navigation**: Marking a baseline does not lock the operator into Compare mode ŌĆö they can navigate freely.

## Architecture

### Comparison Engine

Comparison analysis is generated by a dedicated subsystem, not assembled during rendering.

```
Baseline Result  ŌöĆŌöÉ
                   Ōö£ŌöĆŌöĆŌåÆ  Comparison Engine  ŌöĆŌöĆŌåÆ  ComparisonAnalysis  ŌöĆŌöĆŌåÆ  Renderer
Candidate Result ŌöĆŌöś
```

The Comparison Engine is responsible for:

* Computing deltas between two results
* Classifying each delta as regression / improvement / neutral / uncertain
* Producing a structured `ComparisonAnalysis` value

The Renderer is responsible for:

* Rendering a `ComparisonAnalysis` to a styled string
* No analysis logic ŌĆö it receives already-computed conclusions

### Comparison Session

A Comparison Session tracks the lifecycle of a single comparison workflow:

```
type CompareData struct {
    PinnedBaseline *model.Result      // baseline survives startRun
    Session        ComparisonSession  // ephemeral workflow state
}

type ComparisonSession struct {
    BaselineIndex int              // index into m.results (or -1 when PinnedBaseline is used)
    CandidateIndex int             // index into m.results
    State          SessionState    // Idle ŌåÆ BaselineMarked ŌåÆ Comparing
    Analysis       *ComparisonAnalysis  // computed once per transition
}
```

The session is the source of truth. The renderer never derives what to compare ŌĆö it receives an already-constructed session.

## Comparison Analysis structure

```go
type ComparisonAnalysis struct {
    Baseline  model.Result
    Candidate model.Result

    Metadata  MetadataDelta       // Status, Latency, URL, Error
    Headers   HeaderDelta         // Added, Removed, Changed headers
    Body      BodyAnalysis        // Segments, sizes, change count

    Flags     []Flag              // Regression, Improvement, Anomaly
    Verdict   Verdict             // Equivalent / Improved / Regressed / Mixed
}

type MetadataDelta struct {
    Status  ValueDelta[int]
    Latency ValueDelta[time.Duration]
    URL     ValueDelta[string]
    Error   ValueDelta[string]
}

type HeaderDelta struct {
    Added    []HeaderEntry
    Removed  []HeaderEntry
    Changed  []HeaderChange
}

type BodyAnalysis struct {
    Segments     []BodySegment
    BaselineSize int
    CandidateSize int
    ChangedLines  int
}

type BodySegment struct {
    Kind SegmentKind  // Equal / Delete / Insert
    Old  []string     // baseline lines (Equal, Delete)
    New  []string     // candidate lines (Equal, Insert)
}

type Flag struct {
    Severity FlagSeverity  // Regression / Improvement / Anomaly / Info
    Field    string        // "latency", "status", "header:Content-Type", etc
    Message  string        // "Latency increased by 230ms"
}

type Verdict int
const (
    Equivalent Verdict = iota
    Improved
    Regressed
    Mixed
)
```

This structure is the single source of truth for the renderer. The renderer receives a `ComparisonAnalysis` and renders each field. It never computes deltas itself.

## v0.9.9 Scope

### In scope

| Component | Description |
|---|---|
| Metadata deltas | Status, Latency, URL, Error ŌĆö with regression/improvement classification |
| Header deltas | Added, removed, and changed response headers ŌĆö sorted, with value diff |
| Body summary | Size change, line-count change, truncated preview showing changed regions (line-based, no semantic diff) |
| Regression coloring | Red for regressions, green for improvements, yellow for anomalies |
| Workflow | Mark from Observe, swap, replace candidate, separate clear from exit, cross-run pinning |
| Discoverability | Ribbon actions, active/marked/pinned markers in Observe list |
| State model | ComparisonSession with lifecycle, Pinned flag for cross-run survival |

### Explicitly deferred (v1.0 or later)

| Feature | Reason |
|---|---|
| Full semantic JSON body diff | Needs its own design (JSON path awareness, huge payload limits, streaming) |
| Binary diff | Specialized requirement, not core to HTTP observability |
| Three-way comparison | Additional complexity not justified by v0.9.9 mission |
| N-way comparison | Would require multi-result session model |
| Statistical latency analysis | Needs multiple samples per result |
| Persisted comparison history | Out of scope for TUI-first release |
| Comparison exports | Out of scope for v0.9.9 |
| AI-generated summaries | v1.0+ capability, needs separate architecture |

## v0.10.0 Scope ŌĆö Explain Comparisons

v0.10.0 is a **rendering release**. The comparison engine is unchanged; the renderer was
rebuilt around a single, invariant information hierarchy so operators understand a
comparison within seconds.

### Stable information hierarchy

Every Compare screen answers four questions in exactly this order:

```text
Verdict      ŌĆö highest-level decision: Equivalent / Improved / Regressed / Mixed
  Ōåō
Why          ŌĆö direction-only sentences derived from flags (no raw values)
  Ōåō
Evidence     ŌĆö grouped raw deltas (Status, Latency, Headers, Body, Errors)
  Ōåō
Details      ŌĆö remaining inspection data (URL, full body diff)
```

Only visual presentation may change between widths. Information ordering never changes.

### Rendering invariants

| Invariant | Enforcement |
|---|---|
| Engine performs no rendering | `comparison.go` imports no lipgloss / bubbletea / ansi packages (`TestV099Architecture_EngineNoRenderImports`) |
| Renderer performs presentation only | `renderCompare` consumes a `*ComparisonAnalysis` and never derives deltas |
| Renderer mutates no state | `TestV0100_CompareRenderDoesNotMutate`, `TestV0100_CompareRendererConsumesAnalysisOnly` |
| Section ordering is stable | `verifySectionOrdering` asserts WHY ŌåÆ EVIDENCE ŌåÆ DETAILS order |
| Responsive layouts preserve order | Same assertion across widths 80ŌĆō150; wide no longer splits columns |

### Behaviour-first tests

`TestV0100_CompareBehaviourScenarios` covers identical, improved, regressed, mixed,
metadata-only, header-only, body-only, status-only, latency-only, error transitions, and
timeout transitions. Each verifies verdict, Why, Evidence, and Details correctness.

### In scope

| Component | Description |
|---|---|
| Single semantic pipeline | One ordered set of sections built once; layouts are presentation only |
| Unified single-column layout | Wide, medium, and narrow all scroll vertically through the same journey |
| Why section | Direction-only sentences with subtle Ō¢╝ / Ō¢▓ / ┬Ę glyphs, no raw values |

## Non-Goals (explicit)

1. **This release does not add a diff library dependency.** Body preview diff uses line-based string comparison in pure Go. A proper diff algorithm (patience diff, LCS) may be added in v1.0.

2. **This release does not change Inspect.** Inspect retains its current rendering and interaction model. Compare stops calling Inspect render functions, but Inspect itself does not change.

3. **This release does not add statistical or multi-sample analysis.** Every Comparison compares exactly two single-shot results.

4. **This release does not implement a comparison history.** Sessions are in-memory only. Pinned baselines survive one startRun cycle but are not persisted to disk.

5. **This release does not change the Observe result list beyond adding comparison markers** (ŌŚå, Ō¢Č, ­¤ōī).

## Certification requirements

Compare-specific certification verifies:

* Mark lifecycle (Observe and Inspect paths)
* Compare lifecycle (enter, swap, replace candidate, exit, clear)
* Cross-run pinning (baseline survives startRun)
* Regression coloring (worse = red, better = green)
* Unified single-column layout across all widths (no column splitting)
* Stable Verdict ŌåÆ Why ŌåÆ Evidence ŌåÆ Details ordering
* No duplicated Inspect rendering in Compare
* Comparison Analysis ŌåÆ Renderer data flow (engine produces analysis, renderer consumes it, no analysis logic in renderer)
